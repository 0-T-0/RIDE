 15  """RIDE Plugin API 
 16   
 17  .. contents:: 
 18     :depth: 2 
 19     :local: 
 20   
 21  Introduction 
 22  ------------ 
 23   
 24  RIDE can be extended with plugins written in Python starting with version 0.16. 
 25  The plugin API went through a major overhaul in the 0.20 release and this  
 26  documentation describes how to write plugins against the new API. After the 
 27  changes in RIDE 0.20, the goal is to limit backwards incompatible changes to  
 28  the minimum at least until the 1.0 release sometime in the spring 2010. 
 29   
 30  Finding plugins 
 31  --------------- 
 32   
 33  Plugins are loaded automatically when RIDE starts up. RIDE will look in the 
 34  following directories for plugins: 
 35   
 36  - ``<RIDE_INSTALLATION_DIR>/site-plugins`` 
 37  - ``<USER_DATA>/plugins`` 
 38   
 39  Location of the ``<USER_DATA>`` directory will vary depending on your platform. 
 40  On Linux and OSX this will be ``~/.robotframework/ride`` while on Windows the  
 41  location is ``%APPDATA%\\RobotFramework\\ride``. 
 42   
 43  Each Python file that is found in these directories is dynamically imported and 
 44  inspected. Every subclass of the `Plugin` class in these files will be 
 45  instantiated as a plugin. This has a few noteworthy consequences: 
 46   
 47  - Common utility code may be distributed as separate files, located in the 
 48    plugin directories. 
 49  - All the top level code in the found Python files is executed as they are 
 50    imported. Beware of the side effects. 
 51  - A Python file may contain more than one plugin. 
 52   
 53  Initialization 
 54  -------------- 
 55   
 56  As stated earlier, every plugin *must* inherit from the `Plugin` base class. 
 57  This class is exposed directly from the `pluginapi` module, similarly as all  
 58  other classes that plugins most often need, and can thus be imported like:: 
 59   
 60      from robotide.pluginapi import Plugin 
 61   
 62  When a plugin class is found, an instance of it will be created. Different 
 63  initialization options are explained in the documentation of the  
 64  `Plugin.__init__` method. If creating an instance of a plugin class fails, an  
 65  error message is shown to the user and the plugin is disabled. 
 66   
 67  Enabling and disabling 
 68  ---------------------- 
 69   
 70  Plugins can control are they enabled by default when they are loaded for the 
 71  first time. Afterwards users can enable and disable plugins from the plugin 
 72  manager, which is available from the menu through ``Tools > Manage Plugins``. 
 73  Plugins' enabled/disabled state is stored into RIDE's settings file and read 
 74  from there when plugins are loaded again later. 
 75   
 76  When the plugin is enabled, the `Plugin.enable` method is called. This is the  
 77  point where the plugin is actually turned on and also the time when possible  
 78  integration into RIDE UI should happen. The `Plugin.disable` method is called 
 79  when the plugin is disabled, and its purpose is to undo whatever was done in 
 80  the `Plugin.enable` method. 
 81   
 82  Creating menu entries and shortcuts 
 83  ----------------------------------- 
 84   
 85  Plugins can create new entries to menus, buttons to the toolbar, and register 
 86  shortcuts using `ActionInfo` objects and the `Plugin.register_action` method.  
 87  Registering actions is thoroughly documented in the `robotide.action` module. 
 88  Notice that all the relevant action classes are exposed also through the  
 89  `pluginapi` module and plugins should import them like:: 
 90   
 91      from robotide.pluginapi import ActionInfo 
 92   
 93  Messaging 
 94  --------- 
 95   
 96  RIDE has a messaging system that allows both sending messages when something 
 97  happens and subscribing to certain messages. Messages sent upon some 
 98  user action, like selecting an item in the tree or saving a file, are the 
 99  main communication mechanism from RIDE to plugins, but sometimes plugins may 
100  also have a need to sent their own messages. Plugins can subscribe to messages 
101  using the `Plugin.subscribe` method. The whole messaging system is documented 
102  in the `robotide.publish` module, but plugins can import the relevant classes 
103  through the `pluginapi` module like:: 
104   
105      from robotide.pluginapi import RideTreeSelection, RideSaved 
106   
107  Settings 
108  -------- 
109   
110  Plugin can store information persistently to RIDE's setting file. The initial 
111  values can be given to the `Plugin.__init__` method and new values saved using 
112  `Plugin.save_setting`. Saved settings can be accessed using direct attribute 
113  access via `Plugin.__getattr__`. 
114   
115  Settings are stored into the setting file under ``[Plugins]`` section into 
116  plugin specific subsections. Settings names starting with an underscore  
117  (currently only ``_enabled``) are reserved for RIDE's internal usage. The saved 
118  settings may look something like this:: 
119   
120      [Plugins] 
121      [[Release Notes]] 
122      version_shown = 'trunk' 
123      _enabled = True 
124      [[Recent Files]] 
125      max_number_of_files = 4 
126      recent_files = [] 
127      _enabled = False 
128  """
